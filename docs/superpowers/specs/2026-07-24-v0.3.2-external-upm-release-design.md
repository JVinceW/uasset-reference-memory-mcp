# v0.3.2 External UPM Package Indexing Release Design

Date: 2026-07-24

## Goal

Publish `unity-asset-reference-mcp@0.3.2` with automatic indexing of active
local Unity Package Manager packages stored outside the Unity project, then
provide a structured author brief and a short Discord-ready changelog directly
in the release handoff.

## Version Decision

Repository ADR 0012 normally classifies a new backward-compatible capability as
a MINOR release, which would make this `0.4.0`. The release owner explicitly
selected `0.3.2`.

Treat `0.3.2` as a one-off release-owner exception:

- record the exception in the durable release plan and final evidence;
- do not edit, weaken, or reinterpret ADR 0012;
- do not treat this exception as precedent for later releases; and
- do not reuse or replace any published npm version or Git tag.

## Release Contents

The release includes the already merged external UPM package indexing feature:

- automatic discovery of local directory dependencies from Unity's
  `Packages/manifest.json` and resolved package lock metadata;
- physical reads from external package directories with canonical stored paths
  such as `Packages/com.company.gameplay/...`;
- package-name validation and bounded discovery warnings;
- embedded, external-local, and matching cache source precedence;
- project/package reference extraction across physical root boundaries;
- package discovery fingerprinting and fresh atomic reconciliation when package
  sources change;
- root-level and descendant ignore behavior; and
- product, story, and completed execution documentation.

No additional product capability is added during release preparation.

## Versioned Surfaces

- Root npm package: `0.3.1` to `0.3.2`.
- Root `package-lock.json`: update its root package version consistently.
- Unity verification package
  `unity/com.jvincew.assetreferencememory/package.json`: remain `0.2.0`.
- SQLite schema: remain version 3.
- MCP and CLI inputs and successful response shapes: unchanged.

The release version commit changes only root npm version metadata plus the
durable release plan/evidence required by the repository Harness.

## Publication Flow

Before version mutation:

1. Confirm local `main` is clean and contains the reviewed feature.
2. Confirm `unity-asset-reference-mcp@0.3.2` is absent from npm.
3. Confirm local and remote tag `v0.3.2` do not exist.
4. Confirm no remote `release/0.3.2` state would be overwritten unexpectedly.

Prepare the release:

1. Create and maintain one plan under `docs/plans/active/`.
2. Change the root npm version to `0.3.2`.
3. Run the complete repository validation matrix.
4. Commit the version and release evidence.

Publish through the repository-owned workflow:

1. Push verified `main` to `origin/main`.
2. Push the verified release commit to `origin/release/0.3.2`.
3. Monitor `.github/workflows/release.yml` until its run completes.
4. Stop without tagging if the workflow fails or npm does not expose `0.3.2`.
5. After npm returns exactly `0.3.2`, create annotated tag `v0.3.2` at the
   release commit and push it.
6. Verify local `main`, `origin/main`, the peeled tag, and the published npm
   version agree.

Do not run `npm publish` locally.

## Validation

Before publication, run:

- root-only Vitest excluding `.worktrees/**`;
- `npm run typecheck`;
- `npm run build`;
- root `npm pack --dry-run`;
- Unity package `npm pack --dry-run`;
- `git diff --check`; and
- final release-scope and worktree-status inspection.

The release workflow repeats install, typecheck, tests, build, and npm
publication on GitHub Actions. Registry verification is authoritative for
publication success.

## Failure and Recovery

- Existing npm version or Git tag: stop before mutation and report the
  collision.
- Unexpected remote release branch: inspect it and stop rather than force-push.
- Local validation failure: do not push; fix within the active plan and rerun
  the complete matrix.
- Release workflow failure: do not tag; preserve the release commit and report
  the failed run.
- npm propagation delay after a successful workflow: poll the registry for a
  bounded period; do not create the tag until the immutable version is visible.
- Tag mismatch: stop and report; never move or replace a published tag.

## Structured Blog Author Brief

After verified publication, provide the author brief directly in the
conversation. It is source material for a blog-writing AI rather than a finished
article. Include:

- three suggested titles;
- intended audience;
- one-sentence thesis;
- the modular-project problem and why project-root-only indexing misses local
  UPM packages;
- the physical-root versus Unity-virtual-path mental model;
- manifest/lock discovery, validation, precedence, warnings, and fingerprint
  reconciliation;
- cross-boundary GUID reference behavior;
- concrete test and release evidence;
- trade-offs and exclusions;
- a suggested article outline;
- a factual checklist; and
- claims the blog AI must avoid.

Keep the brief problem-first rather than organizing it as commit history or a
changed-file list.

## Discord-Ready Changelog

After verified publication, print a short Markdown message directly in the
conversation. Do not create a changelog file.

The message must:

- begin with `v0.3.2`;
- state that external local UPM packages are now indexed automatically;
- mention canonical `Packages/<name>/...` paths and cross-package dependency
  tracing;
- mention safe refresh when package sources change;
- stay concise enough to paste into Discord; and
- avoid internal implementation detail, unverified metrics, or unreleased
  claims.

## Compatibility and Scope

This release does not add manual scan-root configuration, Unity Editor
dependencies, package downloads, filesystem watchers, package mutation, schema
changes, or new public MCP tools.

Blog and Discord content are conversational handoff outputs. Only the release
specification, execution plan/evidence, and version metadata are repository
artifacts.
