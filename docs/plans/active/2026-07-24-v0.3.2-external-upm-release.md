# v0.3.2 External UPM Package Indexing Release Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Publish `unity-asset-reference-mcp@0.3.2`, tag the verified release commit, and deliver a structured blog-author brief plus a short Discord-ready changelog directly in the conversation.

**Architecture:** Prepare one narrowly scoped root npm version commit, validate it locally, and publish only through the repository `release/*` GitHub Actions workflow. Treat registry visibility as the publication gate, tag only afterward, then close the durable plan and provide conversational content handoffs.

**Tech Stack:** Git, npm, GitHub Actions, PowerShell, Markdown.

- Release root npm package as exactly `unity-asset-reference-mcp@0.3.2`.
- Record `0.3.2` as a one-off release-owner exception; do not edit or reinterpret ADR 0012.
- Never reuse, replace, force-update, or move an existing npm version or Git tag.
- Keep `unity/com.jvincew.assetreferencememory/package.json` at `0.2.0`.
- Keep SQLite schema version 3 and all public MCP/CLI inputs and successful response shapes unchanged.
- Do not add product behavior during release preparation.
- Publish through `.github/workflows/release.yml`; never run local `npm publish`.
- Do not create or push `v0.3.2` until npm returns exactly `0.3.2`.
- Stop rather than force-push if `origin/release/0.3.2` already exists unexpectedly.
- Print the structured blog-author brief and Discord changelog in the conversation after verified publication; create no changelog file.
- Preserve unrelated worktrees and uncommitted state.

## Risks And Recovery

- **Version-policy exception:** keep it release-local and explicit; ADR 0012
  remains authoritative afterward.
- **Existing immutable target:** stop before mutation; never replace version or
  tag.
- **Unexpected remote release branch:** stop; inspect ownership and SHA; never
  force-push.
- **Validation failure:** do not publish; fix and rerun the full matrix.
- **Workflow failure:** do not tag; preserve run URL and failure evidence.
- **Registry delay:** bounded polling only; tag remains blocked until npm is
  authoritative.
- **Evidence commit after tag:** tag stays on the published release commit;
  final `main` may contain the later plan-completion commit.
