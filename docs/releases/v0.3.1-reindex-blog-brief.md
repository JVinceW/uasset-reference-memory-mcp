# v0.3.1 Reindex Reliability: Author Brief

## Suggested Titles

- Reindexing Unity Assets by Identity, Not by Path
- Why a Unity Asset Index Needs Its `.meta` Files
- Making Asset Graph Refreshes Reliable Through GUID-First Reconciliation

## Audience And Thesis

**Audience:** Unity teams, technical artists, build and tooling engineers, and
developers who depend on an asset-reference graph during refactors or
Addressables work.

**Thesis:** Reliable Unity reindexing starts by treating the asset and its
`.meta` as one GUID-identified unit, then publishing a rebuilt graph only when
the result is safe to trust.

## Problem Summary

Open with the limits of path and file searching: a path answers where a file is
now, not whether it is still the same Unity asset after a move, rename, or
extension change. An index can also go stale after asset or `.meta` edits,
making dependency answers quietly unreliable.

Use Addressables migration as the concrete pressure case. Groups and entries
are generated ownership data that must follow group-asset changes, rather than
being left behind as stale rows. Explain that Unity identity lives in the GUID
inside a sibling `.meta`; the asset file and `.meta` must therefore be handled
as one identity-bearing unit.

## Failure Cases That Motivated The Work

- A `.meta` changes while the asset file does not, but a file-only timestamp
  check misses the update.
- A GUID-preserving move is mistaken for a deletion plus a new asset, breaking
  the conceptual continuity of incoming references.
- A new GUID appears at the old path, and an index silently retargets the old
  asset's references instead of exposing the replacement.
- Duplicate metas make identity ambiguous; choosing one result would make the
  graph non-deterministic.
- A legacy schema-3 index stores uppercase GUID identities while current
  references are lowercase, causing a false replacement on the next refresh.
- Addressables group rows survive a changed or removed group asset and no
  longer describe the project on disk.

## Solution Narrative

Frame the change as reconciliation rather than a faster directory walk:

1. Scan only complete asset/`.meta` pairs and calculate an effective timestamp
   from the newer sibling.
2. Canonicalize GUIDs to lowercase and reject duplicates before any database is
   published.
3. Reconcile current and prior rows by GUID first, with paths treated as
   mutable metadata.
4. Re-extract changed sources and other sources affected by changed target
   types or restored targets.
5. Refresh the Addressables rows owned by the group assets involved.
6. Build a temporary schema-3 database and atomically replace the previous
   good index only after a successful run.
7. Detect a pre-canonical uppercase schema-3 index and perform one compatible
   fresh rebuild rather than attempting an in-place multi-table migration.

## Trade-Offs And Boundaries

- Incremental mode favors speed through timestamps; `force: true` supplies the
  guaranteed-freshness path when timestamps may have been preserved.
- Schema version remains 3. The release does not widen successful MCP or CLI
  response shapes.
- The concurrent `.meta` read/stat race remains deferred. A `.meta` can change
  after it is read but before it is statted. The recovery is to stabilize the
  project and rerun, optionally with force; no retry mechanism is introduced.
- The release does not add analytics, watchers, or Unity callbacks. Refreshing
  stays explicit and agent-managed.

## Factual Checklist

- Describe GUID as the stable Unity identity and path as mutable metadata.
- State that normal `index_project` is incremental and `force: true` rebuilds
  from current project contents for guaranteed freshness.
- State that duplicate GUIDs fail before publication; do not imply automatic
  GUID repair.
- State that same-path replacements produce a warning and leave old references
  unresolved rather than silently retargeting them.
- State that generated Addressables rows are refreshed from their owning group
  assets.
- State that the database schema remains 3 and publication is atomic.
- Describe concurrent writes as a rerun/force recovery case, not as a solved
  retry feature.
- Do not claim watchers, Unity callbacks, analytics, package publication, or a
  schema upgrade.
